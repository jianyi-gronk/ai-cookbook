# 1. 提示注入防御

## 1.1 基础介绍

- 提示注入是 Agent 最核心的安全威胁，分为 直接注入 和 间接注入
  - 直接注入
    - 用户在输入中直接嵌入恶意指令，意图覆盖 Agent 原有的系统指令
  - 间接注入
    - 通过污染 Agent 读取的外部数据源（ 如 网页、文档、邮件 等 ）来隐蔽操控
    - 根本原因在于 LLM 无法在单一上下文中区分可信用户指令与不可信外部数据（ Context Mixing 问题 ）

## 1.2 核心防御措施

- 上下文隔离与规划验证
  - PlanGuard 框架引入隔离的 Planner，仅从用户指令中派生合法操作参考集，通过层级验证机制先强制阻止未授权工具调用，再验证参数偏差是良性格式差异还是恶意劫持，在 InjecAgent 基准上将攻击成功率从 72.8% 降至 0%
- 运行时工具调用边界拦截
  - ClawGuard 在每次工具调用边界强制执行用户确认的规则集，从用户声明的目标中自动推导任务特定的访问约束，在不修改模型或架构的前提下拦截所有三种注入通道（ Web/本地内容、MCP 服务器、技能文件 ）
- 动态规则与注入隔离
  - DRIFT 框架通过安全规划器构建最小功能轨迹和参数检查清单，动态验证器监控计划偏差，注入隔离器检测并屏蔽记忆流中的冲突指令，实现对控制层和数据层的双重约束
- 输入层防护
  - 在输入侧部署提示注入检测 guardrail，识别 “Ignore previous instructions”、“Developer mode activated” 等常见注入模式并阻止

# 2. 越权工具调用防护

## 2.1 基础介绍

- Agent 的工具调用权限是其执行能力的核心，也是最大的攻击面
- 一个典型的越权问题是“能力过度供给”
  - Agent 默认暴露所有可用工具（ 15+ 种 ）给每个会话，即使只是做一个文档摘要任务
  - 攻击者可利用工具调用权限和 “自动执行” 特性，形成 “合法工具链” 组合攻击，实现恶意目标

## 2.2 核心防御措施

- 最小权限的动态能力治理
  - Aethelgard 框架通过四层自适应治理强制执行最小权限原则——能力治理层按会话动态限定 Agent 可见的工具范围，安全路由层在每次工具调用执行前用混合规则 + 微调 LLM 分类器拦截，RL 学习策略层基于审计日志训练最小可行技能集，实验显示摘要任务实现了 73% 的工具减少和 100% 危险工具消除
- 运行时策略执行引擎
  - Agent OS（微软 Agent Governance Toolkit）作为无状态策略引擎，在每次 Agent 操作执行前以亚毫秒级延迟进行拦截和策略评估，支持 YAML 规则、OPA Rego 和 Cedar 策略语言，实现类似操作系统内核的权限控制
- 工具调用中间件拦截
  - ProClaw 作为 OpenClaw 的安全 wrapper，拦截 Agent 的每个工具调用并在执行前执行策略——工具白名单、文件系统权限、网络域控制、速率限制、时间策略，并支持对高风险工具调用进行人工审批
- 能力-行动分离架构
  - ShisaD 在 LLM 和外部系统之间运行持久化守护进程，LLM 只负责提出行动建议，运行时决定实际执行什么，每次工具调用都经过策略执行、污点追踪和审计。

# 3 敏感信息泄露防护

## 3.1 基础介绍

- Agent 系统中的隐私风险具有 “跨阶段传播” 特性
  - 原始用户内容会流入检索查询、记忆写入、工具调用和日志，传统的脱敏管道仅关注文档边界，无法应对这种跨阶段传播

## 3.2 核心防御措施

- 推理前提示词中介
  - BodhiPromptShield 在推理前检测敏感片段，通过类型化占位符、语义抽象或安全符号映射进行路由，并将还原操作延迟到授权边界，实现传播感知的中介和还原时机控制
- PII 检测与掩码
  - 在发送到 LLM 之前检测 SSN、信用卡号、邮箱、电话等敏感信息，可选择直接阻止或将其掩码处理（替换为星号），既保护隐私又不完全破坏上下文理解
- DLP 引擎
  - ProClaw 集成 Presidio + spaCy NER、正则模式匹配、凭证检测和内容脱敏功能，对工具输出进行凭证清理（ 8 种正则模式 + 精确匹配脱敏 ）
- 日志中的敏感数据保护
  - Agent 框架应确保敏感数据仅在显式启用时才被记录，通过 OpenTelemetry 进行日志记录和遥测时需特别保护

## 4. 内容审核

## 4.1 基础介绍

- 内容审核需覆盖输入侧（恶意提示、越狱尝试）和输出侧（有害内容、幻觉），且需在 Agent 的多步执行过程中持续进行，而非仅在聊天边界

## 4.2 核心防御措施

- Pre-LLM + Post-LLM Guardrails
  - Pre-LLM 防护在输入发送到模型前运行，处理 PII 检测、敏感数据阻止和提示注入检测；Post-LLM 防护在模型返回响应后运行，进行幻觉检测、毒性检测、工具和操作验证、输出格式合规检查
- 自校正循环
  - 当 Post-LLM guardrail 检测到问题时，不直接向用户返回失败，而是将标记的问题反馈给 LLM 进行针对性修正，Agent 重试后修正输出再次通过 guardrail
- 外部内容审核 API：集成专业内容审核服务对输入和输出进行毒性、偏见、仇恨言论等检测，结合人机协同机制实现人工监督/干预
- 平台级内容过滤器
  - Azure AI Agent Service 等在平台层面强制执行内容过滤器以确保安全和合规，需设计解决方案使 Agent 在检测到敏感意图时能正确调用自定义工具而非进入异常状态

## 5. 权限隔离

## 5.1 基础介绍

- 权限隔离是 Agent 安全架构的基础，需要在多个层面实现：用户级、会话级、Agent 间、工具级
- Agent 本质上构成具有跨应用操作权限的 “通用代理”，权限失控可能导致系统被完全接管

## 5.2 核心防御措施

- 分层治理架构（LGA）
  - 四层框架——L1 执行沙箱隔离、L2 意图验证、L3 零信任 Agent 间授权、L4 不可变审计日志。实验显示四层协同运作实现 96% 的拦截率，总 P50 延迟约 980 ms，其中非判断层仅贡献约 18 ms
- 沙箱与事务性回滚
  - AI Agent Sandbox 提供安全的代码执行、应用交互和敏感数据处理隔离环境。故障容忍沙箱框架通过基于策略的拦截层和事务性文件系统快照机制，以约 14.5% 的性能开销实现 100% 高风险命令拦截率和 100% 失败状态回滚成功率，优于容器的初始化开销或商业 CLI 的交互摩擦
- 动态执行环
  - Agent Runtime 引入受 CPU 特权级启发的动态执行环（execution rings），将不同风险等级的操作映射到不同隔离级别，并为多步事务提供 saga 编排和紧急终止开关
- 子 Agent 权限交集
  - ProClaw 实现了子 Agent 安全机制，子 Agent 权限必须 ⊆ 父 Agent 权限，防止权限升级
- Agent 间身份与信任
  - Agent Mesh 通过去中心化标识符（ DID，Ed25519 签名 ）、Agent 间信任协议（ IATP ）和 0-1000 的动态信任评分（ 五个行为层级 ），实现 Agent 之间的安全通信和信任管理
