# 1. 背景

- Agent 在执行任务时会接触到大量用户数据，包括 原始输入、检索结果、工具返回内容、记忆写入等，这些数据在任何环节都可能被明文记录或意外流出
- 根本风险：跨阶段传播
  - 问题表现：用户的原始敏感内容（ 如身份证号、邮箱、商业机密 ）不会停留在对话界面，而是会流入检索查询、记忆写入、工具调用参数和系统日志等多个下游阶段
  - 攻击原理：传统的脱敏管道仅关注文档边界（如对存储的文件进行加密），但 Agent 的“数据流动”是动态、多路径的——一次工具调用的返回结果可能在下一次推理时被作为上下文重新注入，形成跨阶段的隐私传播链条
  - 典型场景：用户在对话中说 “我的身份证号是 1234567890”，该信息被 Agent 写入长期记忆，随后在另一场会话中被不加脱敏地检索并输入给 LLM，最终出现在第三方日志系统中
- 与传统数据安全的差异
  - 传统系统：数据流向是静态、可预测的（ 用户 → 数据库 → 应用 ），边界清晰
  - LLM Agent：数据在 LLM、工具、记忆模块、日志系统之间动态流转，任何环节的疏漏都可能导致隐私数据 “穿透” 整个系统

# 2. 防御思路

- 核心理念：从 “点状脱敏” 转向 “传播感知的全链路防护”。不再仅对输入或输出单独处理，而是在 Agent 的整个数据生命周期中追踪和控制敏感信息的流动
- 三大关键拦截点
  - 推理前：在用户输入和工具返回内容进入 LLM 上下文之前，检测并替换敏感片段
  - 工具调用时：对工具的输出进行实时脱敏，防止敏感数据通过工具链向外扩散
  - 日志与存储：在数据写入日志、记忆或数据库之前，确保敏感信息已被脱敏或加密

# 3. 核心防御措施

## 3.1 推理前提示词中介

- 思路：在 LLM 推理之前，对提示词中的敏感信息进行“占位符替换”，将还原操作延迟到授权的安全边界之后执行
- 具体方案（以 BodhiPromptShield 框架为例）
  - 敏感片段检测：识别用户输入和上下文中包含的 PII、机密信息等敏感片段
  - 三种中介策略
    - 类型化占位符：将敏感信息替换为带类型标签的占位符（如 [EMAIL]、[PHONE]），保留语义提示但不泄露具体内容
    - 语义抽象：将具体值替换为抽象描述（如“用户邮箱”→“一个工作邮箱”），在保护隐私的同时维持任务可执行性
    - 安全符号映射：将敏感信息映射为不可逆的安全符号，仅授权边界可还原
  - 延迟还原：原始数据在沙箱或可信执行环境中才被还原，LLM 本身从未接触明文
- 优点：从源头切断敏感数据流入 LLM 的路径，传播感知能力强
- 缺点：对检测精度要求高，过度脱敏可能破坏上下文理解，影响任务完成质量

## 3.2 PII 检测与掩码

- 思路：在数据进入 LLM 或工具链之前，通过规则和 AI 模型识别敏感信息，并进行掩码处理（如替换为星号或通用标识符）
- 具体方案：
  - 检测范围：支持身份证号（SSN）、信用卡号、邮箱、电话、IP 地址、API 密钥、护照号等常见 PII 类型，可配置按国家/地区定制（如 NHS、护照号）
  - 处理方式
    - 直接阻止：检测到敏感信息时直接拒绝请求，返回错误提示
    - 掩码处理：将敏感值替换为 **** 或 [REDACTED]，保护隐私的同时保持上下文结构。
  - 典型实现：agno 框架的 masking 功能在发送到 LLM 前将 PII 替换为星号，使 Agent 理解上下文但不暴露实际敏感信息
- 优点：实现简单，速度极快，适合作为第一道防线
- 缺点：基于模式匹配的方式容易被变体绕过（如格式化变化、大小写变化）；过度掩码可能破坏提示语义，影响模型理解

## 3.3 DLP 引擎集成

- 思路：将企业级数据防泄漏（DLP）能力嵌入 Agent 运行时，对工具输入输出进行深度扫描和脱敏
- 具体方案（以 ProClaw DLP Engine 为例）
  - Presidio + spaCy NER：集成微软开源的 Presidio 框架，利用命名实体识别（NER）模型和正则表达式，识别文本中的姓名、信用卡号、电话、地址等 50+ 种实体类型
  - 多层级检测：同时使用正则模式匹配、NER 模型、凭证检测（8 种正则 + 精确匹配）三层机制，提高检测覆盖率
  - 内容脱敏：对工具输出进行凭证清理，将检测到的敏感值替换为安全占位符，脱敏后的内容才被返回给 LLM 或用户
  - 本地执行：Presidio 所有检测和脱敏均在本地环境完成，敏感数据不离开基础设施
- 优点：检测能力强，覆盖面广，支持多语言，可扩展自定义识别器
- 缺点：部署相对复杂，对非结构化文本的识别可能存在漏报或误报，需持续调优

## 3.4 日志与遥测脱敏

- 思路：在可观测性数据（日志、追踪、指标）写入存储或发送到外部系统之前，对敏感字段进行脱敏处理
- 具体方案
  - 日志脱敏：Agent 框架应确保敏感数据仅在显式启用时才被记录。OpenClaw 实施双通道脱敏——输入参数脱敏和输出内容脱敏，对命令参数和返回结果中的敏感信息进行统一清理
  - OpenTelemetry 集成：通过 OpenTelemetry Collector 的 attributes processor 组件，在遥测数据采集阶段自动屏蔽敏感字段（如信用卡号、API 密钥、用户邮箱），再发送到 Loki、Tempo 等后端
  - 审计日志防篡改：ProClaw 使用哈希链（hash-chained）防篡改事件日志，即使日志被访问也难以修改历史记录。
- 优点：覆盖易被忽视的可观测性管道，防止敏感数据通过运维侧泄露
- 缺点：需在系统设计阶段提前规划，后期补救成本较高

## 4. 局限

- 跨阶段传播的追踪难度
  - 当前防御多聚焦于单点检测，对于敏感数据在“输入 → 记忆 → 检索 → 工具 → 输出”链条中的跨阶段流动，缺乏端到端的追踪能力。BodhiPromptShield 虽提出了传播感知中介的概念，但仍处于学术原型阶段
- 脱敏与任务质量的两难
  - 过度脱敏（尤其是语义抽象或掩码）会破坏上下文的完整性，导致 LLM 无法准确理解任务意图。如何在保护隐私和保持可用性之间取得平衡，仍是一个开放问题
- 多 Agent 场景的组合泄露
  - 多个 Agent 协作时，单个 Agent 的响应可能不含敏感信息，但多个 Agent 的响应聚合后可能暴露敏感模式（组合式隐私泄露）。当前防御对此类威胁几乎无覆盖
- 第三方工具链的不可控性
  - Agent 调用的外部 API 或第三方工具，其内部如何处理和记录数据无法完全控制。即使本地做了完善的脱敏，数据仍可能在第三方侧被明文存储
