# 1. 背景

- Agent 在执行任务时会接触到大量用户数据，包括 原始输入、检索结果、工具返回内容、记忆写入等，这些数据在任何环节都可能被明文记录或意外流出
- 根本风险：跨阶段传播
  - 问题表现：用户的原始敏感内容（ 如身份证号、邮箱、商业机密 ）不会停留在对话界面，而是会流入检索查询、记忆写入、工具调用参数和系统日志等多个下游阶段
  - 攻击原理：传统的脱敏管道仅关注文档边界（ 如对存储的文件进行加密 ），但 Agent 的 “数据流动” 是动态、多路径的，一次工具调用的返回结果可能在下一次推理时被作为上下文重新注入，形成跨阶段的隐私传播链条
  - 典型场景：用户在对话中说 “我的身份证号是 xxx”，该信息被 Agent 写入长期记忆，随后在另一场会话中被不加脱敏地检索并输入给 LLM，最终出现在第三方日志系统中
- 与传统数据安全的差异
  - 传统系统：数据流向是静态、可预测的（ 用户 → 数据库 → 应用 ），边界清晰
  - LLM Agent：数据在 LLM、工具、记忆模块、日志系统之间动态流转，任何环节的疏漏都可能导致隐私数据 “穿透” 整个系统

# 2. 防御思路

- 核心理念：从 “点状脱敏” 转向 “传播感知的全链路防护”。不再仅对输入或输出单独处理，而是在 Agent 的整个数据生命周期中追踪和控制敏感信息的流动
- 关键拦截点
  - 推理前：在数据进入 LLM 上下文之前（ 例如用户输入和工具返回内容 ），检测并替换敏感片段
  - 工具调用时：对工具的输出进行实时脱敏，防止敏感数据通过工具链向外扩散
  - 日志与存储：在数据写入日志、记忆或数据库之前，确保敏感信息已被脱敏或加密

# 3. 核心防御措施

## 3.1 推理前敏感信息处理

- 思路：在数据进入 LLM 上下文之前，对敏感信息进行检测和替换，从源头防止隐私泄露。根据任务需求，可选择原地掩码（ 基础方案 ）或占位符延迟还原（ 高级方案 ）
- 具体方案
  - 检测范围与实现
    - 支持身份证号、信用卡号、邮箱、电话、API 密钥、护照号等常见 PII 类型
    - 可通过正则表达式、NER 模型或企业级 DLP 引擎实现。例如集成 微软 Presidio + spaCy NER，利用三层机制（ 正则模式匹配 + NER 实体识别 + 凭证精确匹配 ）识别 50+ 种实体类型，所有检测均在本地完成，敏感数据不出基础设施
  - 两种处理模式
    - PII 掩码
      - 将检测到的敏感值直接替换为 **** 或 \[REDACTED]，或直接阻断请求
      - 实现简单、速度快，但会破坏上下文语义，导致 LLM 无法理解原信息类型
      - 适用场景：纯文本摘要、分类、情感分析等无需知晓 PII 具体值的任务
    - 提示词中介
      - 占位符替换：将敏感信息替换为带类型标签的占位符（ 如 \[EMAIL]、\[PHONE] ），或进行语义抽象（ 如 “用户邮箱 xxx” → “一个工作邮箱” ）。LLM 知晓信息类型但不知具体内容
      - 延迟还原：原始数据仅在沙箱或可信执行环境中被还原，LLM 全程不接触明文
      - 优点：在保护隐私的同时保持任务可执行性，Agent 仍可正确调用工具
      - 适用场景：需要执行具体操作的任务，如发送邮件、查询用户订单
- 优点：从源头切断敏感数据流入 LLM 的路径，传播感知能力强
- 缺点：对检测精度要求高；过度脱敏（ 尤其是掩码 ）可能破坏上下文，影响任务完成质量

## 3.2 日志与遥测脱敏

- 思路：在可观测性数据（ 日志、追踪、指标 ）写入存储或发送到外部系统之前，对敏感字段进行脱敏处理
- 具体方案
  - 日志脱敏
    - Agent 框架应确保敏感数据仅在显式启用时才被记录
    - OpenClaw 实施双通道脱敏，即输入参数脱敏和输出内容脱敏，对命令参数和返回结果中的敏感信息进行统一清理
  - OpenTelemetry 集成
    - 通过 OpenTelemetry Collector 的 attributes processor 组件，在遥测数据采集阶段自动屏蔽敏感字段（ 如信用卡号、API 密钥、用户邮箱 ），再发送到 Loki、Tempo 等后端
  - 审计日志防篡改：ProClaw 使用哈希链防篡改事件日志，即使日志被访问也难以修改历史记录
- 优点：覆盖易被忽视的可观测性管道，防止敏感数据通过运维侧泄露
- 缺点：需在系统设计阶段提前规划，后期补救成本较高

## 4. 局限

- 跨阶段传播的追踪难度
  - 当前防御多聚焦于单点检测，对于敏感数据在 “输入 → 记忆 → 检索 → 工具 → 输出” 链条中的跨阶段流动，缺乏端到端的追踪能力。BodhiPromptShield 虽提出了传播感知中介的概念，但仍处于学术原型阶段
- 脱敏与任务质量的两难
  - 过度脱敏（ 尤其是语义抽象或掩码 ）会破坏上下文的完整性，导致 LLM 无法准确理解任务意图。如何在保护隐私和保持可用性之间取得平衡，仍是一个开放问题
- 多 Agent 场景的组合泄露
  - 多个 Agent 协作时，单个 Agent 的响应可能不含敏感信息，但多个 Agent 的响应聚合后可能暴露敏感模式（ 组合式隐私泄露 ）。当前防御对此类威胁几乎无覆盖
- 第三方工具链的不可控性
  - Agent 调用的外部 API 或第三方工具，其内部如何处理和记录数据无法完全控制。即使本地做了完善的脱敏，数据仍可能在第三方侧被明文存储
