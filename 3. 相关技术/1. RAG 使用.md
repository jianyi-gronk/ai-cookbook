# 1. 基础介绍

## 1.1 概念

- RAG（ Retrieval-Augmented Generation，检索增强生成 ）是一种结合 **信息检索** 与 **大语言模型生成** 能力的技术框架，旨在解决大模型常见的 “幻觉”、知识无实时性、缺乏外部知识等问题
- 核心思路：在让大模型回答问题之前，先从一个可靠的外部知识库中检索相关的信息，把这些信息作为 “参考资料” 一起交给大模型，让它基于这些资料来生成更准确的答案

## 1.2 工作原理

- 可以分为三步，**检索 -> 增强 -> 生成**
- 检索（ Retrieve ）
  - 当用户提出一个问题时，系统首先会将这个问题转换成一种可计算的向量形式
  - 然后用这个向量去一个预先构建好的知识库（ 比如公司内部文档、最新的网页、法律法规库等 ）中搜索（ 这个知识库里的每一段内容也都被预先转换成了向量 ）
  - 系统通过计算向量之间的相似度，找到与用户问题最相关的 k 条 文本片段，例如 Top 3 或 Top 5
- 增强（ Augment ）
  - 系统将上一步检索到的相关文本片段，与用户的原始问题，按照一个特定的提示模板组合起来，形成一个增强后的提示
  - 这个提示的典型结构是
    - `“请根据以下参考资料回答用户的问题。参考资料：[检索到的文本1]、[文本2]... 用户问题：[用户的问题]”`
- 生成（ Generate ）
  - 系统将增强后的提示发送给大语言模型
  - 大模型在阅读了 “参考资料” 后，根据其中的信息来生成一个有理有据、内容翔实的答案，而不是凭空想象

## 1.3 核心优势

- 解决幻觉问题：模型不再凭空生成答案，而是有据可依，答案的准确性和可信度大幅提升
- 知识实时更新：大模型训练一次成本极高。RAG 只需更新外部的知识库（ 如添加一篇新文档 ），模型就能立刻使用最新知识，成本极低
- 处理私有/小众知识：企业或个人可以将自己的内部文档、数据库作为知识库，让通用大模型变成这个领域的专家
- 提高可解释性：模型在给出答案的同时，可以引用它依据的 “参考资料” 来源，方便用户核查和信任

# 2. 向量库

## 2.1 向量

- 背景
  - 计算机原本只能精确匹配字符串，比如查找包含 “设计稿” 三个字的文本，但无法理解 “生成 UI 界面” 和 “自动产出高保真原型” 其实意思相近
- 定义
  - 把一段文本（ 或图像、音频 ）的 “含义” 转换成在高维空间中的位置（ 向量 ），把 语义相似性 转化为 空间距离，从而能够计算 “意思是否相近”
  - 本质上是一个 浮点数数组，比如 `[0.12, -0.34, 0.56, …, 0.78]`，通常有几百到几千个数字
- 举例（ 在聚合网站搜索 ai 产品 ）
  - 产品 A 的描述：“从文本描述生成 Figma 设计稿” → 向量 `vA`
  - 产品 B 的描述：“AI 自动创建 UI 界面布局” → 向量 `vB`
  - 用户查询：“生成设计稿” → 向量 `vQ`
  - 尽管用词不同，`vQ` 与 `vA`、`vB` 的距离都很近，但跟 “数据库备份工具” 的向量距离很远

## 2.2 生成向量

- 生成向量 是指使用预训练的 嵌入模型（ Embedding Model ）将原始文本映射为固定维度的向量。嵌入模型经过大量语料训练，能够把语义相近的文本映射到向量空间中相近的位置
- 这个过程通常称为 “向量化” 或 “嵌入”，常用的嵌入模型有 `BAAI/bge-large-zh`（ 中文 ）、`text-embedding-3-small`（ OpenAI ）、`intfloat/e5-large-v2`（ 英文 ）等
- 工作原理
  - 输入一段文本，如 “从文本描述生成 Figma 设计稿”
  - 模型将文本切分成 token（ 词 或 子词 ），并通过多层神经网络（ 如 Transformer ）计算每个 token 的上下文表示
  - 最后输出一个固定维度的向量（ 例如 768 维、1536 维 ），这个向量就是整段文本的 “语义指纹”
- 注意：入库和检索时，必须使用同一个嵌入模型来转换，否则向量空间不一致，相似度计算无意义

## 2.3 向量检索

- 向量检索，也称为语义检索或近似最近邻搜索
  - 给定一个查询向量，在一个向量集合中找出与之最相似的 K 个向量的过程
  - 这里的 “相似” 通常用 余弦相似度、欧氏距离 或 点积 来度量
  - 目标是在可接受的精度损失范围内，实现比暴力穷举快得多的搜索速度
- 通常使用 ANN 搜索方法，和传统暴力搜索方法的对比
  - 暴力搜索（ 精确搜索 ）
    - 将查询向量与库中每一个向量逐一计算相似度，然后排序取 Top K
    - 优点：结果精确，100% 召回
    - 缺点：当向量数量达到数万甚至百万级时，计算量过大（ O(N) ），每次搜索可能需要几百毫秒到几秒，无法满足在线服务需求
  - 近似最近邻搜索（ ANN ）
    - 通过预先构建索引结构（ 如 HNSW、IVF、PQ ），只探查向量空间中可能与查询相近的一小部分区域
    - 优点：速度快（ 通常 <10ms ），可扩展至亿级数据
    - 缺点：召回率略低于 100%（ 例如 95% ），但实际应用中通常足够
- 常见 ANN 算法
  - HNSW（ 分层可导航小世界图 ）
    - 构建多层图结构，上层跳跃式搜索，下层精细搜索
    - 查询速度快、召回率高，但内存占用较大
  - IVF（ 倒排文件索引 ）
    - 对向量空间进行聚类，查询时只搜索最近的几个聚类
    - 内存友好，适合超大规模
  - PQ（ 乘积量化 ）
    - 将向量压缩为短编码，用查表近似距离
    - 大幅压缩内存，适合存储受限场景

## 2.4 向量库

- 向量库 是专门为 存储、索引 和 检索高维向量 而设计的数据库系统，核心功能如下
  - 向量存储：保存每个向量的浮点数数组及其关联的元数据，如 产品 ID、名称、描述、链接、标签 等
  - 向量索引：内置 HNSW、IVF、PQ 等 ANN 索引算法，无需手动实现，支持索引构建与增量更新
  - 相似度检索：给定查询向量，快速返回 Top K 最相似的向量及其元数据，支持余弦相似度、欧氏距离等度量
  - 元数据过滤：在向量检索的同时，支持对标量字段进行过滤（ 如 `category = "设计工具"` ），实现 “先过滤再相似度排序” 或 “先相似度再过滤”
  - 数据管理：支持向量的插入、删除、更新、批量导入，以及数据持久化、备份、恢复
  - 高可用与扩展：分布式部署、副本机制、分片，支撑亿级向量和高并发查询
- 如果项目使用 PostgreSQL，可以通过引入 pgvector 扩展去简单实现

# 3. 增强

## 3.1 基础介绍

- 在 RAG 流程中，增强 是连接 检索 与 生成 的关键桥梁。它的核心任务是将检索到的相关文本片段与用户原始问题，按照特定的策略组合成一个结构化的 Prompt，从而让大语言模型能够基于可靠的参考资料生成答案
- 举例，在 AI 产品聚合网站搜索产品，典型结构如下（ 提示 + 参考资料 + 用户输入 + 预期结果 ）

  ```
  请根据以下参考资料回答用户的问题。如果参考资料中没有相关信息，请直接说“不知道”，不要编造答案。

  参考资料：
  [1] Galileo AI：从文本描述生成可编辑的 Figma 设计稿，属于设计稿生成阶段。
  [2] Uizard：将手绘草图或截图自动转换为高保真原型，支持设计稿生成。
  [3] Figma AI：提供智能布局、内容生成等辅助设计能力。

  用户问题：生成设计稿的 AI 产品有哪些？

  请按相关性列出产品名称并简要说明其功能。
  ```

## 3.2 优化思路

- 增强与向量库的协作
  - 向量库负责高效召回语义相关的文本片段，而增强环节负责将这些片段组织成 LLM 易于理解的提示。两者配合的关键点
    - 元数据传递：向量库返回的每个片段应附带足够的元数据（ 产品名、链接、分类等 ），以便增强环节构造结构化的参考资料。
    - 片段顺序：通常按相似度降序排列，但也可根据业务规则（ 如产品热度、更新时间 ）调整顺序，引导 LLM 的注意力
- 上下文压缩
  - 检索到的文本可能冗长或包含噪声
  - 通过 提取式摘要（ 抽取关键句子 ）或 生成式摘要（ 用小型模型重写 ）压缩上下文，减少 token 消耗，提高 LLM 聚焦度
- 重排序
  - 向量检索返回的 Top K 个片段，可能并非按相关性严格排序
  - 在增强前，使用更精细的 交叉编码器（ 如 BAAI/bge-reranker-base ）对片段重新打分，只保留最相关的 Top N（ 例如 N=3 ）送入 LLM，这能显著提升答案精度
- 查询改写
  - 用户原始查询可能模糊或口语化
  - 在检索前（ 或作为增强的一部分 ），用 LLM 将查询改写为更符合知识库用语的形式。例如：用户问“有没有能快速做界面的？”→ 改写为“生成 UI 界面设计稿的 AI 工具”。改写后的查询用于检索，检索结果再与原始查询一起增强
- 多查询融合
  - 生成多个查询变体（ 例如 3~5 个 ），分别检索后合并结果去重，再统一送入增强提示。这样可以提高召回率，避免遗漏相关产品
- 动态片段选择
  - 根据用户问题的复杂度，动态决定检索并送入 LLM 的片段数量。简单问题只需 1~2 个片段，复杂问题可能需要 5~10 个片段。这可以平衡效果与成本

## 3.4 注意事项

- Token 限制
  - 大语言模型的上下文窗口有限（ 如 4K、8K、128K ），增强提示的总长度不能超过模型的最大输入长度，对于超长上下文，需要采用滑动窗口或递归检索
- 提示注入风险
  - 如果检索到的参考资料中包含恶意指令（ 例如 “忽略之前的指令” ），可能影响 LLM 的行为，应对检索内容进行过滤或转义
- 成本控制
  - 每次调用 LLM 都按 token 计费。通过压缩参考资料、减少不必要的指令，可以降低使用成本

# 4. GraphRAG

## 4.1 基础介绍

- GraphRAG（ Graph-based Retrieval-Augmented Generation，基于图的检索增强生成 ）是一种在传统 RAG 基础上引入 知识图谱 的增强框架，通过 AI 生成的知识图谱来增强 AI 驱动的问答系统
- 它将非结构化文本中隐含的实体和关系显式化、结构化，使得系统能够通过图检索定位到与问题相关的结构化子图，为大语言模型提供更丰富、更具上下文逻辑的背景信息

## 4.2 知识图谱

- 知识图谱 用 节点 表示实体（ 如产品、设计师、功能、阶段 ），用 边 表示实体之间的关系（ 如 “属于”、“使用”、“位于…之后” ）
- 在 GraphRAG 的语境下，知识图谱通常是从非结构化文本中 自动构建 的，例如微软的 GraphRAG 系统使用 LLM 从非结构化数据中自动抽取实体和关系，构建知识图谱

## 4.3 工作原理

- GraphRAG 的整体流程可概括为 构建 → 检索 → 增强 → 生成 四个阶段
- 第一阶段：知识图谱构建（ 离线 ）
  - 文本分块与预处理：将原始文档在逻辑边界（ 章节、段落 ）处切分，同时保留出处信息
  - 实体与关系抽取：使用 LLM 或专门的 NER（ 命名实体识别 ）+ RE（ 关系抽取 ）模型，从文本中识别实体（ 人、组织、产品、事件、概念等 ）和关系（ works_for、depends_on、located_in 等 ）
  - 图谱构建与去重：将抽取到的实体和关系存入图数据库，合并同义词和别名，规范化实体名称
  - 社区检测与摘要生成：运行社区检测算法（ 如 Leiden 算法 ）将紧密相关的实体聚类成 “社区”，并为每个社区生成摘要。这些摘要成为后续处理全局查询的检索目标
- 第二阶段：混合检索（ 在线 ）
  - 当用户提问时，GraphRAG 不依赖单一的向量检索，而是采用多种策略的组合
    - 本地邻域检索：从与查询相关的种子实体出发，沿图边扩展 K 跳，检索子图
    - 社区级别检索：检索与查询意图相关的社区的摘要，用于回答全局性问题
    - 向量检索回退：仍使用嵌入相似度检索相关的孤立文本块，作为补充
- 第三阶段：增强与生成
  - 将检索到的结构化子图、社区摘要和文本块一并打包为 LLM 的上下文，生成带有出处的最终答案

## 4.4 与传统向量 RAG 对比

- 知识表示
  - 传统 RAG：文本块 → 向量（ 孤立的语义点 ）
  - GraphRAG：实体 + 关系 → 知识图谱（ 互联的结构 ）
- 检索方式
  - 传统 RAG：相似度匹配（ Top-K 文本块 ）
  - GraphRAG：图遍历 + 邻域检索 + 社区摘要
- 擅长场景
  - 传统 RAG：单跳、事实型问答（ “退款政策是什么？” ）
  - GraphRAG：多跳推理、跨文档汇总、关系密集型查询
- 多跳推理能力
  - 传统 RAG：弱，依赖语义相似度拼凑
  - GraphRAG：强，沿图边显式追踪实体关系链
- 全局理解能力
  - 传统 RAG：弱，只能看碎片
  - GraphRAG：强，社区摘要提供 “鸟瞰式” 视角
- 答案可解释性
  - 传统 RAG：低，仅返回相似度分数
  - GraphRAG：高，可展示推理路径和实体关系
- 实现复杂度
  - 传统 RAG：低，嵌入模型 + 向量库
  - GraphRAG：高，需要图构建、实体抽取、社区检测管道
- 索引成本
  - 传统 RAG：低，单次嵌入调用
  - GraphRAG：高，每个文本块多次 LLM 调用，例如 实体抽取、关系抽取、摘要生成
- 查询成本
  - 传统 RAG：低，一次向量检索
  - GraphRAG：高，可能需要多次图遍历 + LLM 调用
- 数据更新成本
  - 传统 RAG：低，重新嵌入受影响块即可
  - GraphRAG：高，文档变更可能触发实体消歧、关系重抽取、社区摘要重新生成

## 4.5 具体使用

- 通常可以 LlamaIndex + Neo4j 两者配合实现 GraphRAG
  - LlamaIndex
    - 负责知识图谱的构建、管理和查询编排
    - 提供了从非结构化文本中抽取实体/关系（ 通过 LLM 或自定义抽取器 ）、将图谱与向量检索结合、以及提供自然语言查询接口的能力
    - 它本身不存储数据，而是调用后端存储来持久化图谱
  - Neo4j
    - 负责图数据的持久化存储和高效检索
    - 原生管理节点、关系、属性，并提供 Cypher 查询语言
    - LlamaIndex 可以将 Neo4j 作为其后端存储，同时利用 Neo4j 的图遍历能力加速关系查询
