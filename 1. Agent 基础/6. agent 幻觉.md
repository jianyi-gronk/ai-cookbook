# 1. 问题描述

## 1.1 根本原因

- 根因是 LLM 的幻觉
- 数据层：有缺陷的训练数据
  - 模型的知识完全来自训练数据，数据有问题，模型自然学不好
    - 错误与偏见：互联网文本本身就包含大量谣言、过时信息、主观偏见。模型在学习语言规律的同时，也忠实地学会了这些错误
    - 知识覆盖不均 ：对于长尾领域、小众知识或非常新的事件，训练数据中可能只有极少量样本，甚至完全缺失。当被问到时，模型只能依据相似领域的模式进行 “合理编造”
    - 信息压缩的必然损失：将 PB 级的互联网文本压缩进几十 GB 的模型参数中，这个过程注定是有损压缩。模型记住的是统计规律和抽象概念，而非精确的事实细节。例如它知道 “珠穆朗玛峰很高，在尼泊尔和中国边境”，但精确到小数点后两位的高度（ 8848.86 米 ），在压缩过程中很容易变成模糊的 “大约 8800 多米”
- 模型层：基于概率的 “猜词游戏”
  - LLM 的根本任务是 最大化下一个词在训练数据分布中的出现概率，而非 陈述事实
    - 目标是 “像人话”，而非 “说真话”：模型的训练目标（ 如交叉熵损失 ）惩罚的是 “语法不通顺” 或 “词汇搭配罕见”，并不直接惩罚 “内容虚假”。因此，一个语法完美、行文流畅的虚假陈述，在模型看来是 “高分答案”
    - 缺乏 “不确定性” 的内在表达：模型没有内置一个 “我不知道” 的状态。当它遇到知识盲区时，它无法选择坦诚承认，而是会继续完成 “造句” 任务。在它的机制里，虚构一个听起来合理的词（ 例如虚构一个人名或日期 ），比输出一个突兀的 “我不知道” 在概率上更符合训练数据的语言流
- 解码层：追求 “创造力” 的参数设定
  - 在让模型生成文本时，会设置一些解码参数（ 如温度 `Temperature`、Top-p 采样 ）。这些参数控制着模型输出的随机性
    - 高随机性 = 高创造力 = 高幻觉风险：当用户为了获得更有趣、不呆板的回答而调高 Temperature 时，模型会更倾向于选择概率稍低的 “冷门词”。这在写诗、写故事时是优势，但在陈述事实时，就可能从 “珠穆朗玛峰” 跳到 “乔戈里峰”，从而导致事实错误

## 1.2 幻觉放大

- Agent 会放大幻觉
- 任务规划：智能体在行动前，需要将人类复杂的指令分解为一系列可执行的步骤，这个过程本身就可能产生幻觉
  - 目标理解幻觉：误解用户的真实意图
  - 意图分解幻觉：将任务分解为错误或不相关的子任务
  - 计划生成幻觉：基于错误假设制定计划，如试图调用不存在的工具或权限
  - 幻觉目标 ：在规划时可能设定一个在逻辑上或物理上根本无法实现的 “幻想状态”，导致后续所有行动都指向海市蜃楼
- 工具执行：Agent的核心能力之一是调用工具。然而，在执行阶段，幻觉可能表现为
  - 参数幻觉：调用工具时填充了错误的参数
  - 功能幻觉：基于模糊或不准确的工具文档，对工具功能产生误解，导致调用方式错误
- 上下文记忆：在多轮交互中，Agent 需要依赖记忆来维持对话的连贯性，但在记忆的 “存储” 和 “提取” 环节，也可能出现偏差
  - 记忆污染：将先前对话中的幻觉信息存入记忆，并在后续推理中作为“事实”提取，形成恶性循环
  - 上下文误用：错误地理解或过度依赖对话历史中的某段信息
  - 信息冲突：当内部知识、外部检索结果和对话上下文三者信息矛盾时，Agent 可能无法做出正确判断
- 多智能体：在多智能体系统中，幻觉的危害会被急剧放大，形成系统性风险
  - 幻觉传播：单个 Agent 的幻觉会被当做正确信息传递给其他 Agent，导致错误在整个系统中扩散
  - 幻觉 “雪球”：一个微小的幻觉在传递过程中被后续 Agent 不断添加自己的 “理解”，像滚雪球一样越滚越大，最终导致系统做出荒谬的决策

# 2. 解决方案

- Agent 幻觉问题，与一般大语言模型的 “事实性错误” 相比，其形态更为复杂，常表现为 “与人类合理预期不一致的行为”，如错误调用工具、误解用户意图或在任务规划中引入不存在的步骤等，这不仅关乎信息真假，更直接影响 Agent 能否可靠地完成任务
- 要解决这一问题，不能依赖单一方法，而需构建一个覆盖 “生成前约束 -> 生成中验证 -> 生成后纠错” 全流程的多层次防御体系

## 2.1 多智能体协作

- 协作模式
  - 共识与辩论
    - 通过多 Agent 间的讨论达成共识，降低幻觉率
    - 例如，Council Mode 框架调度多个异构模型并行解答，再由 共识模型 进行综合；“苏格拉底式辩论” 则通过批判性提问，提升事实准确性
  - 稽查与协作
    - 通过引入 “督查” 或 “内部自省” 机制来发现错误
    - 例如，HalMit 框架从外部监控 Agent 的行为；InEx 框架则让 Agent 通过 “自我反思” 先审视自身推理，再与其他 Agent 协作验证
- 适用场景
  - 高价值、高风险的任务，如金融分析报告、复杂法律文书起草、医疗诊断建议等

## 2.2 检索增强生成

- RAG 的核心思想是：先检索，后生成
- 具体思路
  - 在 Agent 回答问题前，先从外部可信知识库（ 如 公司文档、数据库、网页 等 ）中检索相关信息
  - 然后将这些信息作为 “参考资料” 提供给 LLM，要求其严格依据这些资料生成答案
  - 例如引入 Web 搜索等工具可将幻觉率降低
- 与 Agent 深度结合
  - 当 RAG 与 Agent 的规划、行动能力结合（ 即 “Agentic RAG” ），系统可以智能地判断何时检索、检索什么、以及如何利用检索结果。这解决了传统 RAG 可能因过度或不当检索引入噪声的问题
- 衍生方法
  - Debate-Augmented RAG（ DRAG ）和 MEGA-RAG` 等方法，分别在检索和生成环节引入多Agent辩论或强化证据筛选，进一步提升生成内容的可靠性
- 适用场景
  - 几乎所有知识密集型任务，如智能客服、企业知识库问答、专业领域的文档辅助等

## 2.3 提示工程与约束

- 这是一种轻量、低成本但非常有效的方法，通过精心设计的指令来引导和约束 Agent 的行为
- 精准定义角色与边界
  - 在 System Prompt 中清晰界定 Agent 的职责、能力和权限
  - 例如，明确告知 Agent：“你是一个只能回答产品问题的助手，如果用户询问天气，请礼貌地表示你无法回答”
- 引导不确定性处理
  - 教导 Agent 在不确定时承认知识的局限
  - 例如，添加指令：“如果你对答案不确定，请直接说 ‘我不知道’，不要编造信息”
- 结构化输出约束
  - 要求 Agent 以特定格式（ 如 JSON、XML ）输出，或使用 “结构化推理字段” 等技术，这不仅能约束生成内容，也便于下游程序解析与校验
  - 另一种高级约束是 “确定性算法”，为 Agent 设定可验证、可执行的规则边界
- 适用场景
  - 场景相对明确、可控的任务，如特定业务流程的引导、数据查询 API 的参数生成等

## 2.4 评估、监测 与 纠正

- 前三种策略主要在生成前和生成中起作用，但仍需构建一套 “防火墙”，在Agent生成内容后进行 评估、监测 和 纠正
- 评估与监控
  - 幻觉检测模型：训练专门的轻量级模型（ 如 HalluGuard ），来检测 Agent 的输出是否有据可查
  - 置信度评分：要求Agent为其输出内容提供置信度分数。分数低的答案可以被标记为“需人工审核”或直接过滤
  - 事实核查工具链：调用外部 API（ 如 Google Fact Check Tools ）对生成内容中的关键事实进行核验
- 纠正与行动
  - 人工审核回路：设计产品流程，让高风险的决策（ 如发送重要邮件、执行交易 ）在执行前需要人工确认
  - 输出改写与拒答：当幻觉被检测到时，系统可以自动尝试修正、要求 Agent 重新生成，或直接拒绝回答
- 适用场景
  - 所有场景，尤其是在生产环境中长期稳定运行的 Agent 应用

## 2.5 优化数据与模型

- 上述策略多为 “流程性” 的解决方案，从根源上减少幻觉的产生同样重要
  - 高质量数据
    - 训练LLM所使用的数据质量直接决定其“知识基础”。确保数据的准确性、时效性和多样性，可以从源头上减少模型自身知识的偏差。
  - 针对性微调
    - 对基础模型进行特定领域的微调，能显著提升其在垂直领域的知识精度和指令遵循能力，减少因“不懂”而导致的编造行为。
- 适用场景
  - 作为所有解决方案的基石，尤其适用于有明确垂直领域和自有数据的业务
