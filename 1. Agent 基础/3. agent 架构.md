# 1. Agent 架构

- Agent 架构通常可以分成四个发展步骤
  - L0：直接动作输出
    - 架构模式：原生 Function Calling
    - 核心特点：模型直接生成工具调用或最终答案，无显示推理轨迹
  - L1：显示推理
    - 架构模式：ReAct
    - 核心特点：交替输出 Thought 和 Action，用自然语言推理指导行动
  - L2：结构化推理
    - 架构模式：Plan-and-Solve，Reflexion
    - 核心特点：增加 预规划，自我反思，长期记忆 等模块
  - L3：**Workflow 编排**（ 多智能体 + 图状态管理 ）
    - 架构模式：Multi-Agent，Graph-based 等，比如 AutoGen，LangGraph
    - 核心特点：多个 Agent 协作或对抗，图状态管理

# 2. 直接动作输出

- 模型接收用户输入和可用工具列表后，通过模型原生 Function Calling 能力，直接输出一个动作（ 即工具调用或最终答案 ），中间不生成任何显式的推理文本，整个决策过程隐含在模型的内部计算中
- 工作流程
  - 用户输入 → 模型（ 内部推理 ）→ 输出动作（ 如调用 search 工具 ）→ 执行动作 → 观察结果 → 模型再次输出动作或答案
- 优点
  - 速度最快，token 消耗最少
  - 实现简单，直接调用模型 API
- 缺点
  - 复杂任务容易出错
  - 难以调试，无法观察模型决策过程
  - 多步推理时容易遗漏中间步骤
- 使用场景
  - 适合单步或两步的简单任务，比如简单的 天气查询，计算器调用 等

# 3. 显示推理

- 模型交替输出 “思考” 和 “动作”，用自然语言写出当前的推理过程，然后根据 “动作” 的观察结果继续 “思考”，直至任务完成
- 工作流程
  - 用户输入 → Thought 1 → Action 1 → Observation 1 → Thought 2 → Action 2 → Observation 2 → ... → Thought n → Finish（ 最终答案 ）
    - Thought：自然语言描述当前状态和下一步计划
    - Action：调用工具或直接给出最终答案
    - Observation：工具执行结果（ 由系统注入 ）
- 优点
  - 高度可解释，便于调试
  - 能处理复杂多步任务
  - 灵活应对动态环境
- 缺点
  - token 消耗大，每步都输出思考
  - 可能陷入无效循环，或者存在无效步骤
  - 推理和行动有时不一致
- 使用场景
  - 需要多步信息检索的任务，如 “比较两款手机哪个好”
  - 需要规划与执行的复杂问题，如 “旅行路线规划”

# 4. 结构化推理

- 在 ReAct 的基础上增加了 预先规划 和 事后反思 等结构化模块

## 4.1 Plan-and-Solve

- 模型在采取任何行动之前，先生成一个完整的多步计划，然后逐步执行计划中的每一步，而不是边想边做
- 工作流程
  - 用户输入 → 生成计划（ 1. ... 2. ... 3. ... ）→ 执行步骤1 → 观察结果 → 执行步骤2 → 观察结果 → ... → 完成
    - 计划可被随时修改（ 如若遇到意外结果 ）
    - 减少了无目的的行动
- 优点
  - 减少中间不必要的思考
  - 任务流程更清晰
- 缺点
  - 初始计划可能错误，导致后续全错
  - 需要更强的规划能力
- 使用场景
  - 需要严格步骤的任务（ 如按照教程配置环境 ）
  - 多工具流水线（ 如 数据清洗 → 分析 → 可视化 ）

## 4.2 Reflexion

- Agent 在执行任务失败后，会生成一段反思文本，存储到长期记忆中，下次遇到类似任务时避免犯同样的错误
- 工作流程
  - 执行轨迹（ 可能失败 ） → 反思生成 → 反思存储（ 记忆库 ）→ 新任务 → 检索相关反思 → 改进行动
    - 反思内容包括：错误原因、正确做法、注意事项
- 优点
  - 能够从失败中学习，持续改进
  - 提高长期任务的鲁棒性
- 缺点
  - 需要额外的记忆存储和检索机制
  - 反思本身也可能出错
- 使用场景
  - 编程调试，如 Agent 记住某类 bug 的修复方法

# 5. Workflow 编排

## 5.1 Workflow

- Workflow（ 工作流 ）是一种由代码预先定义的、固定顺序的任务编排模式，与 Agent 区别如下
  - 决策权
    - Workflow 的决策者是 开发者，代码固定流程
    - Agent 的决策者是 模型，自主推理决定
  - 执行路径
    - Workflow 的执行路径是 预定义、可预测、固定分支
    - Agent 的执行路径是 动态生成、不可完全预测
  - 适用场景
    - Workflow 适合标准化作业、审批流、多阶段报告生成
    - Agent 适合开放式任务、探索性操作、复杂博弈
  - 可控性
    - Workflow 可控性高，每一步可审计、可重试
    - Agent 可控性低，模型可能误判、循环或偏离目标
- 在实际应用中，Wrokflow 和 Agent 通常融合使用：Workflow 作为骨架固化主干流程，Agent 在子节点内部负责局部自主推理与工具调用

## 5.2 图状态管理

- 本质：Agent + Workflow 的融合架构，即 Workflow 管流程，Agent 管执行
- 核心思想：将多 Agent 协作抽象为一张有向图，用节点、边和全局状态实现复杂任务编排
  - 节点：Agent 或工具的执行单元
  - 边：固定边（ 强制流转 ）或条件边（ 动态分支 ）
  - 状态（ State ）：全局共享的数据对象，在节点间传递并更新
- 工作流程
  - 开始 → 规划 Agent → 任务分解 → 执行 Agent → 检查 Agent → 条件分支 → 通过则输出，否则转回规划 Agent → ... → 完成
- 优点
  - 可处理单 Agent 难以完成的超长链路复杂任务
  - 角色分离，每个 Agent 可独立优化，以应对不同场景
  - 流程主干固化，避免纯 Agent 自由探索的失控风险
- 缺点
  - 设计门槛高，需同时定义 Agent 行为与图拓扑
  - 状态同步与通信开销大
