# 1. Prompt Engineering

## 1.1 背景

- 在以 ChatGPT、DeepSeek 为代表的第一阶段 GPT 产品中，大语言模型的能力被封装为一种更高效率的知识库、数据库与搜索引擎替代品
- 这一阶段，AI 应用的核心任务是 问答，一切工作的重心都围绕着一个目标展开：如何让模型更准确地理解用户意图，并输出更符合预期的答案
- 由此，当时研究和实践的中心便落在了 提示词（ Prompt ）上

## 1.2 核心方法

- Prompt Engineering 并非简单地 “写一句更有效的话”，它解决的核心问题是：在模型参数固定不变的前提下，如何优化输入表达以最大化输出质量
- 它是一套关于如何通过构造输入文本来控制模型输出分布的系统方法论，还包括测试、评估和迭代
- 在该阶段，部分方法如下
  - 格式化设定
    - 为模型明确设定身份、背景、语气与行为边界（ 例如 “你是一位资深系统架构师” ），以此限定语义空间，约束生成方向
  - 少样本学习
    - 通过在 Prompt 中预先提供 1~N 个高质量的 “输入-输出” 范例，使模型在推理时进行上下文模仿，显著提升特定任务下的格式遵循度与准确率
  - 思维链（ Chain-of-Thought, CoT ）
    - 要求模型在给出最终结论前，先输出一段隐式的推理过程，如 “让我们一步步思考”
    - 该方法通过将复杂问题分解为中间步骤，极大提升了逻辑推理与数学计算类任务的表现
  - ReAct 模式（ Reasoning + Acting ）
    - 将推理过程与行动步骤交织编排，例如 “思考 -> 行动 -> 观察 -> 再思考”
    - 此方法使模型开始展现出初步的任务拆解与执行规划能力，可视为从纯粹问答向 Agent 概念演进的开端

## 1.3 存在限制

- Prompt Engineering 的能力边界也极为明确。仅凭优化提示词，无法突破以下天然限制
  - 不能凭空给模型补进私域知识
  - 不能让模型知道外部实时发生了什么
  - 不能天然解决跨轮对话的持续记忆
  - 不能代替权限控制、执行环境和错误恢复机制
- 对 LLM 的期待不是一个高级的数据库查询式的问答，而是希望它能处理上述事情。实际上，是在期待一个 Agent，转向了能够自主规划、操作环境、多步执行的更高层级形态

# 2. Context Engineering

## 2.1 背景

- Prompt Engineering 解决了 “如何问” 的问题，却无法解决 “模型不知道” 的问题
- 随着应用场景从通用闲聊走向企业级生产力工具，暴露出一个核心矛盾：模型的训练数据是过去时的、公共的、静态的，而用户需要的是实时的、私有的、动态的信息
- 用户不再满足于模型仅凭 “脑内记忆” 作答，而是期待模型能够
  - 引用公司内部刚刚更新的文档
  - 结合前 50 轮对话中提及的个人偏好
  - 知晓当下的实时天气、股价或新闻事件
  - 明确知晓自己可以调用哪些外部能力及其使用方式
  - 基于刚才工具调用的返回结果继续推理
- 由此，研究和实践的中心从 输入表达 转向 信息供给，即 Context Engineering

## 2.2 定义范畴

- 凡是会进入模型视野、影响其下一步决策的信息，都可以当作 Context
  - 提示词：内化系统预设指令中的角色设定、语气规范与行为边界
  - 用户输入：准确解析当前请求中的显性意图与隐含约束
  - 工具定义：明确知晓自己可以调用哪些外部能力及其使用方式
  - 工具返回结果：基于工具执行后的真实反馈进行动态调整与后续推理
  - 历史对话：回溯并利用完整交互记录中的关键上下文线索
  - 检索出的知识片段：即时调取从外部知识库中匹配到的相关文档内容
  - 长短期记忆：在持久化的用户画像与会话临时状态之间灵活切换信息焦点
  - 当前任务状态：记住并维护多步执行流程的当前进度与中间产出
- 但注意
  - Context Engineering 仅负责
    - 向模型提供可用工具的定义（ 告知模型 “你可以调什么、怎么调” ）
    - 接收工具执行后的结果，并将其注入上下文窗口供模型后续推理
  - **真正发起调用、执行代码、访问网络 等动作，并不属于 Context Engineering 范畴**

## 2.3 核心方法

- Context Engineering 解决的核心问题是：在模型推理窗口有限的前提下，如何从海量的外部数据与历史交互中，精准检索、压缩并注入最相关的上下文信息
- 它并非简单的 “复制粘贴资料”，而是一套关于 **信息检索**、**记忆管理**、**动态注入** 的系统工程
- 在该阶段，部分方法如下
  - RAG（ 检索增强生成 ）
    - 将私域知识库（ PDF、Confluence、代码仓库 等 ）预先切分并转化为向量索引
    - 用户提问时，先检索出语义最相似的 Top-K 文档片段，作为背景知识拼接至 Prompt 中
    - 本质是给模型配了一本可随时查阅的 “活页夹”
  - 向量检索的深度优化（ HyDE / ColBERT ）
    - HyDE（ 假设性文档嵌入 ）
      - 先让模型根据问题凭空生成一段假设性答案，用这段假设文本去做向量检索，从而弥合 “用户提问词汇” 与 “文档实际用词” 之间的语义鸿沟
    - ColBERT（ 延迟交互 ）
      - 不将长段落压缩为单个向量，而是保留每个 Token 的独立向量，在检索阶段进行词级精细匹配，解决长文本检索中的细节丢失问题
      - 缺点则是存储开销爆炸
        - 传统方法中，一篇 2000 字的文档，只存 1 个 1024 维的向量（ 约 4KB ）
        - Token 级方法，同一篇文档有 2000 个 Token，需要存 2000 个 向量（ 约 8MB ）
  - 对话记忆管理（ Memory Management ）
    - 滑动窗口
      - 仅保留最近 N 轮对话原文，超出部分直接丢弃
    - 摘要压缩
      - 通过背景线程持续将长对话历史压缩为一段百字以内的结构化摘要
    - 向量化长期记忆
      - 将历史对话切片存入向量库，用户提及 “上次说的那个方案” 时，通过语义检索动态召回相关片段
  - 动态上下文注入
    - 根据用户身份（ VIP / 试用期 ）、时间（ 工作日 / 节假日 ）、地理位置，动态选择注入不同的业务规则 Prompt 片段
    - 实现 同一套模型能力，千面千境的服务体验
  - 工具定义注入（Tool Definition Injection ）
    - 在调用模型前，将可用工具的名称、功能描述、参数 Schema 以结构化格式（ 如 JSON Schema ）注入系统 Prompt
    - 模型据此理解自身可调用的外部能力边界，并生成符合规范的调用指令
    - 本质是将系统能力描述转化为模型可见的 “菜单”
  - 工具执行结果注入（ Observation Injection ）
    - 当模型发出工具调用指令并经 Harness 层执行后，将执行结果（ 如 API 返回数据、数据库查询结果、代码执行输出 ）作为新的上下文消息追加进对话历史
    - 模型得以基于真实世界反馈进行下一步推理或修正
    - 本质是将环境反馈转化为模型可见的 “观察值”

## 2.4 存在限制

- Context Engineering 虽解决了“模型知道什么”，但其能力边界同样清晰。仅凭信息注入，无法突破以下限制
  - 无法强制模型遵守格式规范：注入再多上下文，模型仍可能在 JSON 末尾多输出一个逗号
  - 无法阻止模型产生幻觉：即便提供了正确的上下文，模型仍可能因概率采样而输出与上下文矛盾的内容
  - 无法执行真实世界的操作：模型能从文档中读出 API 调用方法，但无法真正去发送那个 HTTP 请求
- 要让 LLM 从 “读过且会说” 跨越到 “听懂且会做”，仅靠信息供给是不够的，这便引出了更为底层的 Harness Engineering

# 3. Harness Engineering

- Agent = 模型 + Harness

## 3.1 背景

- Prompt Engineering 解决了 “如何问”，Context Engineering 解决了 “模型不知道”，但它们都无法解决一个根本问题：**模型既不会真正动手执行操作，其输出行为也难以强制约束**
- 在实际生产环境中，以下场景屡见不鲜
  - 要求模型输出 JSON，它却在前言中加了一句 “好的，这是您要的数据： ”
  - 提供了正确的上下文，模型却仍然编造了一个不存在的数字
  - 模型推理出了正确的 API 调用步骤，但无人真正去执行它
- 用户对 LLM 的期待已经从 “能说会道” 升级为 “听话照做”，这要求系统必须对模型的输出和行为拥有强制性的、不可违背的控制力
- 由此，研究和实践的中心从 信息供给 转向 行为约束与系统调度，即 Harness Engineering

## 3.2 核心方法

- Harness Engineering 解决的核心问题是：如何不通过提示词去软性引导，而是用代码强行框定模型的行为边界，对模型的输入输出进行物理层面的拦截、修正与编排，使其行为严格符合预期边界
- 在该阶段，部分方法如下
  - 约束解码（ Constrained Decoding ）
    - 在模型生成每个 Token 时，实时修改概率分布，将不符合指定格式（ 如 JSON Schema、正则表达式 等 ）的 Token 概率强行置为负无穷
    - 效果：模型物理上无法输出非法字符，JSON 格式正确率从 95% 提升至 100%
    - 实现手段：基于 有限状态机 或 上下文无关文法 的语法掩码
  - 函数调用与工具编排（ Function Calling / Tool Orchestration ）
    - 模型不再直接生成最终答案，而是输出结构化的工具调用指令，如 { "tool": "search", "query": "北京天气" }
    - Harness 层拦截该指令，执行真实的代码或 API 调用，并将结果以观察值形式重新注入上下文
    - 本质是模型负责决策，系统负责执行的分离架构
  - Agent 循环状态机（ ReAct Loop / Agentic Loop ）
    - 构建一个围绕模型的无限循环控制器
      - 调用模型生成 “思考 + 行动”
      - 解析行动指令，调用对应工具
      - 将工具返回结果拼接入上下文
      - 再次调用模型继续推理，直至模型输出最终答案
    - 该循环将单次推理扩展为多步规划与执行的完整工作流
  - 沙箱执行（ Sandbox Execution ）
    - 模型生成的代码（ 如 Python、SQL ）绝不直接在宿主机执行
    - 而是运行在隔离的容器、轻量虚拟机（ 如 Firecracker MicroVM ）或 WebAssembly 运行时中
    - 同时施加严格的资源限制（ CPU 时间、内存上限、网络白名单 ），防止恶意或错误代码造成破坏
  - 输出护栏（ Output Guardrails / Content Moderation ）
    - 在模型输出送达用户之前，设置一道独立的检查层
    - 检测到敏感词、隐私信息（ 如身份证号 ）或越狱企图时，物理切断返回流，返回预设的安全提示
  - 推理性能加速（ Inference Optimization ）
    - 推测解码
      - 用小模型快速生成草稿 Token，大模型并行验证，一次性接受多个正确 Token，提升生成速度
    - 上下文缓存（ Prompt Caching / KV Cache Reuse ）
      - 针对长文档问答场景（ 如阅读几十万字的项目文档 ），模型首次处理文档时，会将长文档部分产生的注意力中间结果暂存于显存
      - 后续提问，文档部分无需重新计算，直接从显存读取缓存结果，仅对新提问部分进行计算
      - 极大降低首 Token 延迟与推理成本

## 3.3 存在限制

- Harness Engineering 虽实现了对模型行为的 “硬约束”，但其边界同样存在
  - 无法提升模型的内在推理能力，约束解码只能保证格式正确，不能把一个笨模型变聪明
  - 系统复杂度急剧上升，引入状态机、工具执行、沙箱等组件后，系统从无状态 API 演变为有状态分布式应用，调试和运维成本激增
  - 安全攻防永无止境，沙箱逃逸、提示词注入绕过护栏、工具调用参数投毒等，需要持续的安全对抗投入
- Harness Engineering 是 LLM 从 “不可控的问答工具” 走向 “生产级基础设施” 的最后一道、也是最关键的一道工序，它用工程确定性包裹住了模型的概率不确定性

# 4. 举例

- 以一个自动化代码 Review 工具的构建过程为例，说明 Prompt、Context、Harness 三者在实际项目中的递进关系与协同方式

## 4.1 纯 Prompt 驱动

- 最初的设计思路非常直接：编写一份详尽的长篇 System Prompt，在其中明确定义角色（ “资深架构师” ）、输出格式（ 结构化 Review 意见 ）以及评估维度（ 代码规范、性能、安全等 ），共计两三百行
  - 现象：实际运行效果呈现显著的 **不稳定性**。同一段代码多次评审，给出的评级可能在 A 与 C 之间随机跳变，规律难以捉摸
  - 诊断：问题并非 Prompt 不够精致，而是模型视野内缺乏做出稳定判断所需的 **上下文依据**。模型只能基于代码本身和通用知识进行猜测，无法获知该 Pull Request 的业务背景、该模块的历史风险记录，以及团队内部的特定编码约定

## 4.2 引入 Context Engineering

- 为解决信息缺失问题，系统引入了 RAG 机制。将代码仓库中的 README、架构设计文档、历史 Issue 及修复记录预先处理为向量索引。当收到 Review 请求时，自动检索并注入最相关的背景信息
  - 效果：评审准确率从约 60% 提升至 85% 左右。模型能够结合业务背景理解代码改动的意图，并引用历史踩坑记录提出针对性建议
  - 局限：RAG 同样存在边界，例如在排查一个隐蔽的内存泄漏问题时，语义检索召回的全是通用性的 “内存管理最佳实践” 文档，意见虽然正确但缺乏针对性，未能直接定位根源

## 4.3 Harness Engineering 介入

- 面对 RAG 无法解决的深层诊断需求，系统进一步为模型配置了 **可执行的工具接口**（ Harness ）。对于上述内存泄漏问题，Harness 层允许模型在隔离沙箱中调用 `valgrind` 等分析工具，并将运行日志作为 Observation 返回
  - 效果：模型基于真实的运行时数据分析，在十分钟内精准定位了泄漏点。这验证了 Harness Engineering 的核心价值：**模型的能力边界不仅取决于其推理水平，更取决于它被赋予了怎样的工具与反馈回路**

## 4.4 总结

- 演进路径
  - Prompt Engineering 负责定调，定义评审标准与输出格式
  - Context Engineering 负责供给，补齐业务上下文与历史经验
  - Harness Engineering 负责赋能，提供真实世界的操作接口与执行反馈
- 三者并非替代关系，而是 **层层递进的工程补全**，共同将模型从 “能评论的读者” 转变为 “能动手的审查员”
