# 1. 评测难点

- 结果导向难以全面评估
  - 缺少过程优劣判断
    - 对同一任务可能存在多种执行路径和最终结果，执行过程在 效率，鲁棒性，资源消耗 等方面可能存在差异，单纯从结果分析无法全面评估
  - 部分成功评估
    - 当 Agent 完成了任务的 80% 时，但是剩下 20% 完成过程中存在误差，如何科学地评估其成功程度等
- 过程评测复杂
  - 过程分析与评估：Agent 评测需要分析整个决策和执行过程，包括 ”思考链“，工具选择逻辑等，评估每个决策步骤是否合理
  - 路径优化评价：不同的优化目标（ 时间，成本，准确性 等 ）可能导致不同的路径选择，“最优路径” 难以定义
- 评测成本大
  - 时间与人力成本大：复杂的 Agent 任务可能包含数十步交互步骤，完整评测需大量时间
  - 自动化实现困难：很多评测环节需要人工判断 Agent 的任务完成质量，自动化落地场景受限

# 2. 评测思路

- 分三个阶段，从关注 “做的对不对” -> “如何做的” -> “整体做的好不好“
- 第一阶段：结果导向评测，黑盒评测，仅关注最终输出结果
- 第二阶段：关注中间过程和决策路径
- 第三阶段：白盒评测，全流程、多维度的综合评估

## 2.1 常见指标

- 任务纬度指标
  - 任务完成度：Agent 成功完成指定任务的比例
  - 任务稳定性：Agent 成功完成相同任务的比例
  - 任务结果效果：Agent 回答或输出内容，即最终答案
  - 任务进度率：Agent 在任务执行过程中的进展程度
- 工具纬度指标
  - 工具调用准确率：Agent 执行每个步骤的准确性，工具调用时是否含冗余步骤
  - 最佳工具选择率：Agent 选择最佳工具的比例
- 逻辑推理
  - 相关性：Agent 执行每个步骤的准确性，工具调用时会含冗余步骤
  - 一致性：Agent 在多轮交互中，保持逻辑一致性的能力
  - 调度效率：Agent 需要最少次数完成任务
- 性能纬度
  - 成本：Agent 处理任务消耗的 Token 数量，以及计算资源（ GPU 等 ）
  - 执行时间：Agent 完成任务所需的时间
- 其他指标
  - 安全性与合规：主要包括 敏感内容生成，越权调用技能 等
  - 对抗与鲁棒性指标：对不同程度噪音，模糊，prompt 工具，工具故障等场景的表现和容错能力
  - 重试次数：Agent 因失败而重新尝试的次数
  - 持续学习与反馈闭环：把在线采集到的真实交互数据用于评估模型退化与自我改进

## 2.2 评测组件

- Trace（ 执行轨迹 ）
  - 是解决 “结果导向难以全面评估” 的关键手段，记录 Agent 从收到任务到结束的完整过程，主要包括 每一步的思考、工具调用、输入输出、中间状态 等
  - 在评测中，trace 是过程分析的基础数据，用于判断 “为什么做了某个决策”，“是否存在无效步骤”，“是否绕了远路”
- State Snapshot（ 状态快照 ）
  - 在 Agent 执行的关键节点（ 如每次与工具交互后、每轮思考结束时 ）保存完整的内部状态（ 内存、变量、对话历史、环境状态等 ）
  - 评测中的作用
    - 支持回放与断点调试：发现某一步出错时，可从快照恢复并重试不同策略
    - 支持路径对比：比较不同 Agent 或同一 Agent 在不同参数下在同一快照点的状态差异
    - 帮助评估 “部分成功”：通过快照可以判断前 80% 的步骤是否正确改变了系统状态，即使最后 20% 失败
- Replay Engine（ 回放引擎 ）
  - 能够基于 trace 和 state snapshot 重新执行 Agent 的行为序列，或让另一个 Agent 从某个快照点继续执行
  - 评测价值
    - 自动化回归评测：对修改后的 Agent 回放相同的任务 trace，检查结果是否一致或更好
    - 过程级评分：将标准答案的 trace 与待测 Agent 的 trace 对齐，比较每一步的决策合理性（ 如工具选择、参数使用 ）
    - 代价可控的复现：对于多步交互的复杂任务，不需要每次真实调用外部 API，而是利用回放引擎模拟环境响应，大幅降低评测成本

## 2.3 评测方法

- 人工排查
  - 判断 agent 输出结果是否准确（ 可以通过人工判断，也可以通过训练评测打分模型来判断 ），以及通过 Trace 分析 Agent 执行过程
- 自动化评测
  - 在简单场景（ 如单轮对话 ），通过预定义结果预期，基于 Agent 工作流的输入和产出（ 包括工具调用等中间信息 ），利用 LLM 的理解和判断能力来评估 Agent 的表现
