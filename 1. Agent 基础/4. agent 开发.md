# 1. 基础 Agent

## 1.1 快速创建 Agent

- create_agent 基于 ReAct 架构，为生产环境设计，具备多项关键特性
  - 标准化的 ReAct 循环：提供了开箱即用、生产就绪的 “思考-行动-观察” 循环
  - 可靠的状态持久化：基于 LangGraph，天然支持任务的检查点、持久化和中断恢复
  - 流式输出：支持逐步流式输出 Agent 的中间思考过程和最终结果
  - 结构化输出：能确保 Agent 的返回结果符合预定义的数据模式
- 常见参数
  - model
    - Agent 的推理引擎
    - 支持传入模型标识符字符串（ 如 "openai:gpt-4o" ），或通过 init_chat_model 等方式初始化的模型实例
  - tools
    - Agent 可以调用的外部工具列表
    - 工具可以是任意函数、API 或数据库查询等
  - system_prompt
    - 为 Agent 设定角色和基本行为准则的系统提示词
  - response_format
    - 定义 Agent 输出的结构化格式，例如指定一个 Pydantic 模型，Agent 将返回符合该模型的数据
  - middleware
    - 这是 create_agent 的核心扩展点
    - 允许插入一个或多个中间件，以横切（ AOP ）的方式无侵入地增强 Agent 的核心能力

## 1.2 常见中间件

- 中间件允许在 Agent 执行的生命周期中的多个关键节点插入自定义逻辑，实现强大的横切关注点管理，提供了 6 个生命周期钩子
  - before_agent
    - Agent 执行开始前触发，通常用于 初始化、加载记忆、权限校验，每次调用仅 1 次
  - after_agent
    - Agent 执行结束后触发，通常用于 清理资源、结果落库、最终状态处理，每次调用仅 1 次
  - before_model
    - 每次模型调用前触发，通常用于 上下文注入、消息过滤、提前终止，每轮循环 1 次
  - after_model
    - 模型返回后、工具执行前触发，通常用于 工具参数修正、日志记录、埋点，每轮循环 1 次
  - wrap_model_call
    - 包裹整个模型调用过程，通常用于 重试机制、缓存、降级、熔断，每轮循环 1 次
  - wrap_tool_call
    - 包裹整个工具执行过程，通常用于 工具重试、权限校验、结果后处理，每个工具调用 1 次
- LangChain 提供了丰富的内置中间件
  - SummarizationMiddleware
    - 功能：在上下文窗口过长时，自动对对话历史进行摘要
    - 使用场景：构建拥有长期记忆的聊天机器人，避免 token 超限
  - HumanInTheLoopMiddleware
    - 功能：在 Agent 调用配置的敏感工具前，暂停执行并请求人工审批
    - 使用场景：需要人工确认的高风险操作，如发送邮件、删除数据
  - PIIMiddleware
    - 功能：在将数据发送给 LLM 前，自动检测并编辑（ 如替换、遮盖 ）个人信息
    - 使用场景：需要遵守数据隐私法规（ 如 GDPR ）的应用
  - ToolRetryMiddleware
    - 功能：当工具调用失败时，自动进行重试
    - 使用场景：处理网络波动或外部 API 临时不可用的情况
  - ModelFallbackMiddleware
    - 功能：当主模型调用失败时，自动切换到备用模型
    - 使用场景：提高 Agent 的鲁棒性，避免因单一模型服务中断而导致应用不可用
  - ModelCallLimitMiddleware
    - 功能：限制 Agent 调用模型的次数，防止陷入无限循环或产生过高的 API 费用
    - 使用场景：在预算有限或需要严格控制执行成本的场景下使用

# 2. 状态图

- StateGraph 由以下几个核心模块
  - 主图：是 agent 的核心，负责协调各个子图的执行，类似总指挥，根据当前状态决定下一步做什么
  - 子图：是完成特定功能的模块
  - 状态管理：智能体通过状态来跟踪任务进度和存储信息
- 以一个最简单的排查错误信息 demo 举例

  ```python
  from langgraph.graph import StateGraph, END

  # 定义状态
  class SimpleAgentState:
    def __init__(self):
      self.input = ""
      self.analysis = ""
      self.solution = ""

  # 节点函数
  def process_input(state):
    print("处理输入:", state.input)
    return { "input": state.input }

  def analyze(state):
    analysis = f"分析: {state.input} 是一个常见问题"
    print(analysis)
    return { "analysis": analysis }

  def generate_solution(state):
    solution = f"解决方案: 针对 {state.input}，建议检查相关配置"
    print(solution)
    return { "solution": solution }

  # 创建状态图
  workflow = StateGraph(SimpleAgentState)
  workflow.add_node("process_input", process_input)
  workflow.add_node("analyze", analyze)
  workflow.add_node("generate_solution", generate_solution)

  # 设置流程
  workflow.set_entry_point("process_input") # 设置起始节点
  workflow.add_edge("process_input", "analyze")
  workflow.add_edge("analyze", "generate_solution")
  workflow.add_edge("generate_solution", END)

  # 编译并执行
  graph = workflow.compile()
  result = graph.invoke({ "input": "服务器连接失败" })
  print("最终结果:", result)
  ```

# 3. 多智能体

- 多智能体系统通常采用以下三种经典协作模式
  - 监督者模式
    - 一个 Supervisor Agent 负责分发任务给多个 Worker Agent，并汇总结果
    - 使用场景：任务可分解为多个独立子任务，如多源信息查询
  - 层级团队模式
    - 多个 Supervisor 形成树状结构，每个 Supervisor 管理一组 Worker
    - 使用场景：超大规模任务，需要分层管理
  - 全对全通信模式
    - 每个 Agent 都可以与其他 Agent 直接通信，通过共享状态协调
    - 使用场景：需要反复讨论、迭代优化的任务，如辩论、代码评审
- 通常可以把 create_agent 返回的 ReActAgent 直接作为节点添加到 StateGraph 中，这样既能复用 ReAct 能力，又能融入图的编排逻辑
